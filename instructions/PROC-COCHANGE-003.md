# PROC-COCHANGE-003: Co-change check (code + docs sync)

> ID: PROC-COCHANGE-003
> Version: 1.0
> Level: **[C] Critical**
> Last Updated: 2026-06-22
> Related: RULE-DOC-010 (docs sync — this procedure implements it), STD-META-001 (ID system), STD-DOC-002 (markdown standard)

> **Status:** ACTIVE. File `guard/scripts/co-change-check.sh` exists.
> Implements RULE-DOC-010 (documentation sync, no code without docs).

## When this procedure fires

Pre-commit hook. Inspects staged files via `git diff --cached --name-only`
and verifies that every code change is accompanied by at least one doc
change in the same commit.

Typical trigger:

```bash
bash guard/scripts/co-change-check.sh           # soft warn (exit 0)
bash guard/scripts/co-change-check.sh --hard    # hard fail (exit 1)
```

Recommended wiring: install as a git pre-commit hook with `--hard` flag.
Platform's `install-hooks.sh` can do this; see §"Integration" below.

## What it does

1. Get list of staged files from `git diff --cached --name-only`
2. Classify each file:
   - **code** (`.py`/`.js`/`.ts`/`.tsx`/`.jsx`/`.sh`/`.go`/`.rs`/etc.)
   - **docs** (`.md`)
   - **tests** (`.test.*`/`.spec.*`)
   - **config** (`.json`/`.yml`/`.toml`/`.ini`)
   - **lockfile** (package-lock, yarn.lock, Cargo.lock, etc.)
3. Decision:
   - No code staged -> PASS (pure-docs/config/lockfile commit)
   - Code + at least one .md -> PASS
   - Code without .md -> VIOLATION (RULE-DOC-010)
4. Print summary, exit 0 (PASS/SOFT-WARN) or 1 (FAIL in --hard mode)

## Exemptions

| File type                         | Exempt? | Reason                                  |
| --------------------------------- | ------- | --------------------------------------- |
| Lockfiles (package-lock, etc.)    | YES     | Auto-generated, no doc meaning          |
| Test files (.test._, .spec._)     | YES     | Tests are themselves a form of docs     |
| Config files (.json, .yml, .toml) | YES     | Naturally flat, no narrative to sync    |
| Pure-docs commits                 | YES     | No code -> rule does not apply          |
| Auto-generated code               | YES     | If header comment marks it as generated |

## Inputs

- `--hard` (optional): exit 1 on violation instead of warning
- `--help` / `-h`: print usage

## Outputs

- Stdout: human-readable classification + decision
- Exit 0: PASS or SOFT-WARN
- Exit 1: FAIL (--hard mode, code without docs)
- Exit 2: usage error

## Calls

None. Pure git-diff inspection.

## Integration with platform hooks

The platform's `install-hooks.sh` script can install this procedure as a
git pre-commit hook:

```bash
# .git/hooks/pre-commit (auto-generated by install-hooks.sh)
#!/usr/bin/env bash
exec bash /home/z/my-project/Z-ai-platform/guard/scripts/co-change-check.sh --hard
```

When integrated, every `git commit` will be blocked if it contains code
changes without corresponding .md changes.

## Limitations (known)

1. **Cannot detect semantic doc-code mismatch.** This procedure only
   checks "is there A .md change" — not "is the .md change relevant to
   the code change." A staged `README.md` typo fix will satisfy the
   check even if the code change is unrelated.
2. **No semantic diff.** Does not detect "code added a new function but
   docs don't mention it." That requires TOOL-VERIFY-001 (verify-docs),
   which is planned but not yet built.
3. **No exemption for hotfix commits.** A production hotfix that needs
   to ship immediately will be blocked in `--hard` mode. Workaround:
   use `git commit --no-verify` (but this also bypasses other hooks).
